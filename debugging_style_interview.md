My company used this type of interview to screen people.

We’d have some REALLY simple code and a unit test that failed.

People who sucked:

    wouldn’t immediately run the unit test

    wouldn’t ask their code buddy any questions (hey, I don’t know this IDE, how do I run the unit test?)

    wouldn’t vocalize their process

    had a meltdown when they couldn’t solve it after 5 mins

People who did well:

    ran the unit tests

    just asked or said if they were stuck (that’s what pair programming is for)

    vocalized their process (“I’m just going to start with running the unit tests and see what fails. Then I’ll take a closer look at the code and go from there..”)

    remained calm when they were stressed
